## 一、文档概述

### 1.1 文档目的

本文档为"识界"旅行足迹地图系统 MVP 版本的产品需求文档（PRD），用于明确一期功能边界、数据需求及用户交互逻辑，作为设计、开发与测试的基准依据。V1.2 版本基于当前实现状态进行修订，补充已实现的城市维度、旅程时间线、个人中心等能力，并明确尚未完成的后台分类/景区管理、4A 景区数据、成就分享等后续规划。

### 1.2 项目背景

随着国内旅游市场的持续升温，越来越多的旅行者希望以可视化的方式记录和展示自己的旅行足迹。本系统以"中国地图点亮"为核心玩法，通过标记 4A/5A 级景区帮助用户可视化旅行成就，激发探索欲望。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 点亮 | 用户标记某景区为"已去过"的状态操作，支持记录具体游览日期 |
| 再次点亮 | 对已点亮景区追加一条新的游览记录，形成多次到访历史 |
| 取消点亮 | 删除该景区最新一条点亮记录；若该景区无剩余记录，则恢复为未点亮 |
| 景区等级 | 国家 A 级旅游景区评级，本系统覆盖 4A 级和 5A 级 |
| 省级行政区 | 包括 23 个省、5 个自治区、4 个直辖市、2 个特别行政区，共 34 个 |
| 城市 | 地级市、地区、自治州、盟等行政区划，用于景区内部分组与筛选 |
| 旅程 | 按时间维度聚合用户的点亮记录与成就解锁记录，形成个人旅行时间线 |
| 旅程时间线 | 以年份/月份为轴，展示用户历次点亮与成就解锁的混合视图 |
| 管理员 | 拥有 `admin` 角色的用户，可进入系统管理后台维护用户与系统配置 |

## 二、产品定位与目标用户

### 2.1 产品定位

一款面向国内旅行爱好者的轻量级足迹记录工具，核心以"中国地图点亮"为载体，通过标记 4A/5A 级景区帮助用户可视化旅行成就，激发探索欲望。

### 2.2 目标用户画像

- **核心用户**：25-45 岁国内旅行爱好者，年均出行 3-6 次
- **次要用户**：旅行规划者，通过景区库了解目的地信息
- **潜在用户**：社交分享型用户，乐于展示旅行成就

### 2.3 MVP 边界

- **范围包含**：
  - 中国省级行政区划地图展示与点亮统计
  - 省份详情页景区浏览、搜索、分类/城市/等级筛选、批量点亮
  - 同一景区多次点亮与取消点亮，支持记录游览日期
  - 城市维度数据展示与筛选
  - 用户注册/登录、JWT 登录态保持、角色权限
  - 个人中心社区资料编辑
  - 旅程时间线（按年份聚合点亮与成就记录）
  - 省份/分类点亮进度可视化
  - 省份探索线、景区达人线、特殊彩蛋成就系统
  - 系统管理后台（用户管理、系统配置）

- **范围排除（后续版本）**：
  - 社交分享（除成就分享卡片预留）
  - 行程规划
  - UGC 内容（用户上传景区/攻略）
  - 商业化功能
  - 4A 级景区完整数据（当前仅覆盖 5A）
  - 后台景区/分类 CRUD 与批量导入（当前通过种子数据维护）

## 三、功能需求

### 3.1 地图展示与统计模块

#### 3.1.1 需求描述

系统首页展示中国地图，用户可直观查看各省份的点亮状态，同时在页面显著位置展示旅行统计数据。地图支持交互式选择省份，右侧可展开"点亮进度 Top5"侧边栏。

#### 3.1.2 功能点清单

1. **地图渲染** —— 使用 MapLibre GL 渲染中国省级行政区划矢量地图，配合 CARTO 底图与 Mapzen DEM 地形晕渲
2. **省份着色** —— 已点亮省份按点亮率使用蓝绿色系渐变着色；未点亮省份显示默认灰色
3. **省份选中** —— 点击省份后高亮显示，并弹出该省简要信息浮层（名称、已点亮/总景区数、点亮率）
4. **省份计数器** —— 页面展示"已点亮 X/34 个省级行政区"（含 23 省、5 自治区、4 直辖市、2 特别行政区）
5. **景区点亮率** —— 展示"已点亮景区 X/Y 个"，其中 Y 为系统当前景区总数，X 为用户已点亮的景区数量（去重）
6. **城市探索统计** —— 展示"已探索城市 X/Y 个"，反映用户点亮景区所覆盖的城市数量
7. **Top5 侧边栏** —— 右侧可展开"点亮进度 Top5"列表，按省份点亮率排序；单击聚焦该省份，双击进入省份详情页
8. **旅行目标** —— 首页支持设置"下一目标省份 + 目标进度 + 期望达成时间"，数据保存在本地存储
9. **世界地图入口** —— 首页提供世界地图占位入口，点击提示"正在积极探索中"

#### 3.1.3 验收标准

- 地图加载时间 < 2 秒（3G 网络环境下）
- 点亮状态变更后，地图颜色在 300ms 内更新
- 统计数据实时同步，与景区点亮操作保持一致
- 未登录用户可正常浏览地图与省份浮层信息

### 3.2 省份景区浏览与点亮模块

#### 3.2.1 需求描述

用户点击地图上的省份或 Top5 列表后，进入该省景区列表页，可浏览该省全部 4A/5A 级景区并批量点亮。列表支持按城市、分类、等级筛选，支持按名称模糊搜索。

#### 3.2.2 功能点清单

10. **景区列表展示** —— 按景区等级（5A/4A）分组展示，组内按名称拼音排序；支持按城市进一步分组
11. **景区信息卡片** —— 每张卡片展示：景区名称、等级标签（5A/4A）、所属分类标签、复选框
12. **复选点亮** —— 用户可通过复选框勾选多个景区，点击"确认点亮"批量标记
13. **日期选择** —— 批量或单条点亮时弹出日期选择器，默认当天，可补录历史游览时间
14. **再次点亮** —— 已点亮景区支持"再次点亮"，追加一条新的游览记录
15. **取消点亮** —— 已点亮景区支持"取消点亮"，删除该景区最新一条点亮记录；无剩余记录时恢复未点亮状态
16. **分类筛选** —— 支持按景区分类（山岳、湖泊、森林公园等）筛选当前省份的景区
17. **城市筛选** —— 支持按城市筛选当前省份的景区
18. **等级筛选** —— 支持按 5A/4A 等级筛选
19. **模糊搜索** —— 支持按景区名称进行模糊搜索，输入关键词实时匹配景区名称
20. **返回地图** —— 提供返回按钮，点击回到首页地图视图

#### 3.2.3 业务规则

- 一个省份只要有一个景区被点亮，该省在地图上即显示为"已点亮"状态
- 取消所有景区的点亮状态后，该省恢复为"未点亮"状态
- 景区点亮操作需用户登录后才可执行
- 同一景区多次点亮时，省份/景区统计按去重计算；旅程时间线保留每次记录
- 4A 级景区数据待补充，当前系统仅初始化 5A 级景区

### 3.3 景区分类管理模块

#### 3.3.1 需求描述

系统支持对景区进行分类管理，分类用于景区筛选和后续统计。MVP 阶段分类信息通过种子数据预置，后台分类 CRUD 与批量修改景区分类功能待后续版本实现。

#### 3.3.2 功能点清单

21. **预置分类** —— 系统预置常见分类：山岳、湖泊、森林公园、历史文化、海岛、古镇古村、现代建筑、动物园/植物园、博物馆、宗教场所
22. **分类 CRUD（后续版本）** —— 后台管理端支持分类的创建、编辑、删除（删除前需校验是否有关联景区）
23. **分类与景区关联（后续版本）** —— 每个景区可关联一个主分类，后台支持批量修改景区分类
24. **分类统计** —— 首页或管理后台展示各分类下的景区数量；分类点亮进度在个人中心以横向进度条展示

#### 3.3.3 验收标准

- 分类名称支持 2-20 个中文字符
- 删除分类时，若该分类下存在已点亮的景区记录，需给出明确提示
- 分类修改后，前端筛选条件实时更新

### 3.4 用户注册与登录模块

#### 3.4.1 需求描述

提供简易的用户账号体系，支持用户注册、登录和基本的个人信息管理。点亮数据与用户账号绑定。系统预置默认管理员账号，用于进入管理后台。

#### 3.4.2 功能点清单

25. **用户注册** —— 支持用户名+密码注册；用户名 3-20 位字母/数字/中文，密码 6-20 位
26. **用户登录** —— 用户名+密码登录；登录后点亮数据与当前账号关联
27. **登录状态保持** —— 使用 JWT Token 本地存储保持登录态，避免频繁重新登录
28. **角色体系** —— 用户角色分为 `user`（普通用户）和 `admin`（管理员）
29. **默认管理员** —— 系统初始化时创建默认管理员账号 `admin / admin123`
30. **个人信息查看** —— 展示用户名、注册时间、已点亮统计、已获得成就
31. **修改密码** —— 支持旧密码验证后设置新密码
32. **游客模式** —— 未登录用户可浏览地图和景区列表，点亮、目标保存、个人中心等操作需登录

#### 3.4.3 业务规则

- 用户名全局唯一，不区分大小写
- 未登录用户可浏览地图和景区列表，但点亮操作需登录后触发
- 每个用户的点亮数据独立存储，互不影响
- 管理员可修改任意用户密码、删除非自身用户

### 3.5 旅程时间线模块

#### 3.5.1 需求描述

为用户提供按时间维度聚合的旅行记录视图，将每次点亮操作与成就解锁记录混合展示，形成个人旅程时间线。

#### 3.5.2 功能点清单

33. **年份聚合** —— 按年份分组展示该年内的点亮记录与成就解锁
34. **混合时间线** —— 同一年份内，按时间倒序展示点亮景区、再次点亮、成就解锁等事件
35. **景区记录展示** —— 每条点亮记录展示景区名称、所属省份、等级、点亮日期
36. **成就解锁展示** —— 成就解锁事件展示成就名称、徽章样式、解锁时间
37. **空状态提示** —— 无记录年份展示引导用户去点亮的空状态

#### 3.5.3 业务规则

- 再次点亮会在时间线中生成独立记录
- 成就解锁记录与用户点亮记录按时间顺序混合排列
- 未登录用户无法查看旅程页

### 3.6 成就与进度系统

#### 3.6.1 需求描述

系统通过可视化进度图表和成就徽章，激励用户持续探索更多省份和景区。用户在个人中心可查看自己的点亮进度和已解锁的成就。

#### 3.6.2 省份点亮进度图

以环形图（Doughnut Chart）展示省份点亮进度，中心显示点亮省份数和总省份数（如 "12/34"）。环形分段按地理大区着色，已点亮段为实心主题色，未点亮段为浅灰色。鼠标悬停各段可查看该大区已点亮/总省份数。

- **触发位置**：个人中心首页、地图页侧边栏
- **更新频率**：每次点亮/取消点亮后实时更新
- **大区划分**：华东 7 省、华南 3 省、华北 5 省、华中 3 省、西南 4 省、西北 5 省、东北 3 省、港澳台 4 区

#### 3.6.3 景点点亮进度图

以横向进度条组（Progress Bar Group）展示各分类下的景区点亮进度。每个分类一行，左侧为分类图标和名称，右侧为进度条（已点亮数/该分类总数），已完成部分用分类主题色填充，未完成部分用浅灰色填充。

- **触发位置**：个人中心、省份详情页侧边栏
- **默认按点亮率从高到低排序**
- **支持点击展开/收起全部分类**
- **进度条末端显示百分比**（如"68%"）和具体数量（如"12/18"）

#### 3.6.4 成就等级体系

#### 3.6.5 特殊成就

特殊成就独立于等级体系之外，通过特定行为触发解锁，增加产品的趣味性和探索感。特殊成就无等级之分，每个为独立徽章。

| 成就名称 | 解锁条件 | 徽章样式 | MVP 实现状态 |
|----------|----------|----------|--------------|
| 旅途起点 | 完成首次点亮任意景区 | 火炬-启程 | ✅ 已实现 |
| 七日新星 | 注册 7 天内点亮 10 个景区 | 流星-限时 | ⏳ 待实现 |
| 夜游神 | 同一天点亮 5 个不同省份的景区 | 月亮-闪电游 | ✅ 已实现 |
| 分类大师 | 某一分类下点亮所有景区 | 奖杯-全收集 | ✅ 已实现 |
| 完美省份 | 某一省份的所有景区全部点亮 | 皇冠-全境 | ✅ 已实现 |
| 5A征服者 | 点亮所有 5A 级景区 | 钻石-五星 | ✅ 已实现 |
| 年度旅人 | 注册满一年且点亮数 ≥ 10 | 日历-周年 | ⏳ 待实现 |
| 坚持旅者 | 连续 30 天每天至少有 1 次点亮 | 火焰-连续 | ⏳ 待实现 |

- 特殊成就展示在成就墙"彩蛋"分区，与等级成就区分展示
- 特殊成就解锁条件不直接展示，仅显示"???"提示语，解锁后方 reveal 具体内容
- 部分特殊成就互斥（如"七日新星"与注册时间相关，过期不可补获）

#### 3.6.6 功能点清单

38. **成就解锁** —— 用户达到成就条件时自动解锁，后端返回 `newAchievements` 列表；前端成就分享与 Toast 提示待补齐
39. **成就展示墙** —— 个人中心展示已获得成就徽章，按省份探索/景区达人/特殊成就分区展示，支持按已获得/未获得筛选
40. **进度可视化** —— 省份和景区点亮进度以图表形式实时展示
41. **成就分享卡片（后续版本）** —— 生成成就解锁分享卡片（图片），展示成就名称、解锁时间、当前进度
42. **特殊成就彩蛋** —— 未解锁的特殊成就显示为"???"占位，解锁后揭示真实名称和徽章
43. **排行榜（后续版本预留接口）** —— 预留成就排行榜接口，后续版本可展示全站点亮排名

#### 3.6.7 验收标准

- 成就解锁延迟 < 1 秒（点亮操作后）
- 进度图表响应时间 < 500ms
- 未解锁成就不暴露解锁后的徽章样式，仅展示灰度轮廓和条件提示
- 成就数据与用户账号绑定，换设备登录后成就进度完整保留

#### 3.7.1 需求描述

个人中心展示用户的旅行统计、点亮进度、成就墙和社区资料。用户可编辑昵称、签名、所在地、旅行标签和头像，打造个性化旅行主页。

#### 3.7.2 功能点清单

## 四、数据需求

### 4.1 景区数据来源

数据采用"爬取官方数据源为主 + 后台手动导入为辅"的双通道方案。初始化时自动导入全国 5A 级景区名录（当前 MVP 阶段），数据维度包括：景区名称、所属省份、所属城市、景区等级（5A）、所属分类、拼音。4A 级景区数据与后台批量导入功能后续版本补齐。

### 4.2 核心数据表

| 数据表 | 核心字段 | 说明 |
|--------|----------|------|
| 省份表 | id, name, code, region | 中国 34 个省级行政区基础信息及所属地理大区 |
| 城市表 | id, name, province_id, type | 地级市/地区/自治州/盟等行政区划 |
| 景区表 | id, name, province_id, city_id, level, category_id, pinyin | 4A/5A 级景区主数据 |
| 分类表 | id, name, sort_order | 景区分类字典表 |
| 用户表 | id, username, password_hash, role, created_at | 注册用户信息及角色 |
| 系统配置表 | key, value, description, updated_at | 站点级配置项 |
| 点亮记录表 | id, user_id, attraction_id, lit_at | 用户与景区的多对多关系，支持同一景区多次点亮 |
| 成就配置表 | id, name, type, level, condition_value, condition_desc, icon, badge_style | 成就徽章定义 |
| 用户成就表 | id, user_id, achievement_id, unlocked_at | 用户已解锁成就记录 |

### 4.3 数据量级估算

- 省级行政区：34 个
- 城市：约 333 个（地级市/地区/自治州/盟）
- 4A 级景区：约 3,000-3,500 家（当前未导入，后续版本补充）
- 5A 级景区：约 340-360 家（当前已导入 367 条）
- 景区分类：预置 10 类，可扩展
- 成就配置：省份线 5 级 + 景区线 10 级 + 特殊成就 8 个 = 23 个成就徽章
- 预计 MVP 阶段用户量：1,000-5,000 人

## 六、待确认问题与风险

### 6.1 已确认决策

- 【点亮含义——已确认】点亮景区 = 标记"已去过"，即用户实际到访过该景区；支持多次到访记录
- 【地图范围——已确认】包含全部 34 个省级行政区（23 省、5 自治区、4 直辖市、2 特别行政区），港澳台在地图上依法合规展示
- 【技术栈——已确认】前端 React 19 + TypeScript + Vite + MapLibre GL；后端 Node.js + Express 5 + TypeScript + better-sqlite3
- 【数据导入——已确认】MVP 阶段通过 `seed.ts` 初始化 5A 级景区数据；4A 数据与后台批量导入后续版本实现
- 【搜索方式——已确认】省份详情页支持按景区名称模糊搜索，实时匹配结果
- 【数据同步——已确认】MVP 阶段只考虑手动触发同步，后续迭代再考虑定时自动同步机制
- 【成就系统——已确认】包含省份和景区双进度图 + 省份探索线 5 级 + 景区达人线 10 级 + 特殊成就 8 个
- 【多次点亮——已确认】支持对同一景区多次点亮并记录每次 `lit_at`，统计按去重计算
- 【社区资料——已确认】社区资料（头像、昵称等）当前仅本地存储，不与账号同步

### 6.2 待实现清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 4A 级景区数据导入 | 高 | 当前系统仅覆盖 5A 景区，需补充约 3,000-3,500 条 4A 数据 |
| 成就解锁 Toast 提示 | 中 | 后端已返回 `newAchievements`，前端需消费并展示 Toast |
| 缺失特殊成就实现 | 中 | 七日新星、年度旅人、坚持旅者 3 个特殊成就待补齐 |
| 后台分类 CRUD | 中 | 分类的创建、编辑、删除及关联校验 |
| 后台景区维护 | 中 | 单条新增/编辑/删除、Excel/CSV 批量导入 |
| 成就分享卡片 | 低 | 生成成就解锁图片卡片 |
| 排行榜接口 | 低 | 全站点亮排名预留接口 |

### 6.3 风险提示

- 【数据风险】景区等级可能动态调整（升级/降级/摘牌），需规划数据更新机制
- 【地图合规】使用中国地图需确保审图号合规，建议使用官方授权地图服务
- 【性能风险】4A+5A 景区总量约 3,500 条，省份详情页列表渲染需考虑虚拟滚动优化
- 【数据一致性风险】社区资料仅本地存储，用户换设备或清空缓存后资料丢失
- 【城市数据风险】当前部分景区 `city_id` 为空，城市分组与筛选效果未完全生效
